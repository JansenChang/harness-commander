# QUALITY_SCORE

## 这个文件是做什么的

用于定义什么样的代码才算高质量，并提供统一的验收标准和自检依据。

## 适合写什么

- 单元测试覆盖要求
- 复杂度限制
- 命名、可读性、可维护性要求
- Review 打分标准

## 推荐用法

- AI 改完代码后，根据这里进行自我评分
- Code Review 时，用这里减少“凭感觉评审”
- 当团队对质量标准不统一时，用这里建立共同语言

## 质量目标

- 默认目标不是“能跑”，而是“可验证、可维护、可复盘、可继续演进”。
- 任何一次实现都必须同时回答四个问题：
  - 功能是否符合产品与协议文档
  - 失败路径是否被验证
  - 改动是否可被后续 AI / 人类快速理解
  - 本次踩坑是否被转化为可复用约束

## 验收维度

### 1. 正确性

- 必须符合 `docs/product-specs/`、`ARCHITECTURE.md`、相关 `docs/` 规范。
- 不允许为了通过测试而绕过产品协议、结果合同或真实失败语义。
- CLI / JSON / 文件产物三者必须共享同一份事实，不允许“文本成功、JSON 漂移、落盘行为不同步”。

### 2. 测试覆盖

- 每个改动至少要覆盖：
  - 1 条成功路径
  - 1 条主要失败路径
  - 1 条边界或回退路径（如果存在）
- 如果问题是回归或 bug，必须补一条能稳定复现该问题的测试。
- 如果问题来自 AI / Codex 的错误实现，必须把“复现用例 + 防复发测试”一起补上，不能只做静默修补。
- 默认优先级：
  - 先单元 / CLI 测试锁合同
  - 再 integration 测试锁真实行为
  - 有包装 / 安装 / 入口链路时补 acceptance

### 3. 可读性

- 命名必须表达业务意图，而不是实现细节。
- 复杂逻辑必须收敛成小函数或清晰的数据结构，不允许靠长函数堆条件分支。
- 新增注释只允许解释“为什么这样做”，不解释显而易见的“代码在做什么”。
- 文档、测试名、错误码、summary 必须能让后续 AI 直接据此推理，不依赖额外口头上下文。

### 4. 可维护性

- 不允许引入平行规则源；已有规则必须回写到对应 `docs/` 文件，而不是散落在代码和对话里。
- 新增约束必须放到可导航的位置：
  - 产品规则 -> `docs/product-specs/`
  - 质量规则 -> `docs/QUALITY_SCORE.md`
  - 可靠性规则 -> `docs/RELIABILITY.md`
  - 技术原则 -> `docs/design-docs/`
  - 轻量 AI 参考材料 -> `docs/references/*-llms.txt`
- 任何“这次先这么写”的临时方案，都必须在 `docs/exec-plans/tech-debt-tracker.md` 留痕。

### 5. 防复发

- AI / Codex 造成的已确认错误，不允许只在代码里悄悄修掉。
- 必须至少完成以下两项：
  - 在对应测试层补上防回归断言
  - 在 `docs/exec-plans/tech-debt-tracker.md` 或完成归档文档中记录：错误现象、根因、修复方式、后续注意事项
- 同类错误第二次出现，应把它升级为团队规则，回写到本文件或 `docs/RELIABILITY.md`，而不是继续靠人记忆。

## 评分规则（10 分制）

- 9-10 分：产品对齐、失败路径完整、测试充分、文档更新到位、后续 AI 可直接接手。
- 7-8 分：主路径和关键失败路径齐全，但仍有少量文档或集成覆盖欠缺。
- 5-6 分：功能基本可用，但测试、失败语义或文档存在明显空洞。
- 0-4 分：只能跑 happy path，或通过临时修补掩盖真实问题，不可接受。

## 最低门槛

- 未补测试的 bug 修复：不合格
- 改了行为但没同步文档：不合格
- 发生 AI / Codex 误修却没留下复盘记录：不合格
- 引入新规则却没有事实源落点：不合格

## AI 自检清单

- 我改动的行为是否能在产品文档里找到依据？
- 我是否覆盖了成功、失败、边界 / fallback 三类路径？
- 我是否把新的约束写回了仓库，而不是只留在对话里？
- 如果这次是 Codex 的错误，我是否留下了可复盘记录和防回归测试？
- 如果三天后另一个 agent 接手，它能只靠仓库文档理解这次改动吗？
