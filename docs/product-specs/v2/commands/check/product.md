# V2 `check` 产品草案

## 当前状态

- draft

## V1 现状

- 已能基于质量、安全、技术信仰和产品相关文档输出审计结果
- 已支持阻断项、warning、未量化规则和默认扫描对象

## V1 缺陷

- `check` 仍偏向静态规则扫描，而不是项目治理总入口。
- 对文档完整性、知识库交叉引用、版本漂移、执行计划漂移的检查还不够强。
- 结果虽然结构化，但还不够面向“下一步应该怎么做”的产品决策。
- 还没有把 `check` 变成真正的“准备进入下一阶段前的门”。

## V2 要解决的问题

- 把 `check` 从规则扫描器升级成项目健康度 / 治理完整性入口。
- 扩展检查对象：不仅检查代码和生成文档，还要检查知识库结构、版本入口、索引完整性。
- 把结果从“发现问题列表”提升到“给 agent / 人类的下一步决策输入”。

## V2 继承的边界

- `check` 继续默认不依赖宿主模型决定最终结果。
- 规则来源、严重级别、位置和建议动作继续需要结构化输出。

## V2 可能推翻的边界

- V1 的默认扫描范围可能过窄。
- `warning` / `failure` 分层也许需要更细粒度的可执行分级。

## 当前开放问题

- `check` 是否应该成为进入 `run-agents`、发布、归档前的统一前置门？
- 是否要显式增加对索引文档、`references/*-llms.txt`、V1/V2 版本切换的检查？
- 是否需要引入项目健康分数，而不只是消息列表？
