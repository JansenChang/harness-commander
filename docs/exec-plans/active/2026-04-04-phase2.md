# 2026-04-04 Harness-Commander 第二阶段完善计划

## Goal

- 基于第一阶段已实现的骨架，完善 Harness-Commander 的完整七类能力
- 实现 `sync`、`distill`、`check` 三个核心命令，补齐产品规格要求
- 完善模板引用机制，使 `init` 命令能够读取外部模板文件
- 增加端到端集成测试，确保命令间协作和完整流程验证
- 引入静态类型检查和代码质量工具，提升代码可靠性

## Context

- 第一阶段已成功实现 `init`、`propose-plan`、`plan-check`、`collect-evidence` 四个核心命令
- 代码架构清晰，分层合理，测试覆盖良好
- 根据对齐报告，当前实现与产品规格对齐度为85%，主要缺失 `sync`、`distill`、`check` 命令
- 需要完善模板引用机制，使 `init` 命令符合关键规则15的要求
- 需要增加更严格的边界检查，确保白名单约束有效

## Business Logic

- `harness sync` 命令应在识别到重大变更时更新 `docs/generated/` 或 `docs/references/` 目录
- `harness distill` 命令应默认调用宿主模型，从文件、文件片段和用户说明生成供下游 LLM / Agent 使用的 `.llms` 结构化上下文包；最终输出结构、路径和状态语义仍由 Harness 控制。
- `harness check` 命令应基于质量、安全和团队信仰文档执行审计，区分阻断项与提醒项
- `harness init` 应使用 `docs/design-docs/init-templates.md` 中的模板内容，而非内置模板
- 所有新命令必须支持统一的 JSON 输出和 dry-run 模式
- 所有命令必须遵循现有的分层架构和错误处理机制

## Scope

### 1. `sync` 命令实现
- 检测数据库结构、迁移文件、公共工具或参考目录的重大变更
- 只更新受影响的 `docs/generated/` 或 `docs/references/` 文件
- 输出变更摘要，包含变更文件列表和变更原因
- 支持 `--dry-run` 模式展示预期变更

### 2. `distill` 命令实现
- 输入一个或多个文件、文件片段和用户说明，输出 `.llms` 结构化上下文包
- 提取摘要、关键关系、参考单元和 Agent 使用指引等机器可读上下文
- 支持文档与源码混合输入
- 默认输出到 `docs/references/`，也允许指定输出路径

### 3. `check` 命令实现
- 基于 `docs/QUALITY_SCORE.md`、`docs/SECURITY.md` 和 `docs/design-docs/core-beliefs.md` 执行审计
- 区分阻断项（必须修复）和提醒项（建议修复）
- 输出规则来源、严重级别、文件位置和处理建议
- 对于阻断项返回非零退出码

### 4. 模板引用机制完善
- 修改 `init` 命令，使其读取 `docs/design-docs/init-templates.md` 中的模板
- 当模板文件缺失时，使用内置默认模板并记录警告
- 确保模板内容包含明确的用途说明、适合内容范围和推荐用法

### 5. 边界检查强化
- 在 `init` 命令中增加运行时白名单校验
- 防止创建未授权目录，如 `src/`、`tests/`、`.venv/` 等
- 当识别到超出白名单的创建请求时直接失败

### 6. 测试与质量提升
- 增加端到端集成测试，验证命令间协作
- 引入 mypy 进行静态类型检查
- 增加代码覆盖率报告

## Exception Handling

- 当 `sync` 命令无法检测变更时，输出明确错误信息
- 当 `distill` 命令无法从输入材料中生成足够的结构化上下文时，输出失败摘要而非不完整 `.llms` 产物
- 当 `check` 命令的规则文档无法解析时，将对应项标记为"未量化"
- 当 `init` 命令的模板文件缺失时，使用内置模板并记录警告
- 当 `init` 命令识别到超出白名单的创建请求时，直接失败并输出超出项清单
- 所有命令的异常都转换为统一的错误消息格式

## Risks

- `distill` 命令的多源输入解析与结构化蒸馏逻辑可能复杂，需要同时处理文档、源码和片段引用
- `check` 命令的规则匹配需要将自然语言规则转换为机器可判断条件
- 模板引用机制可能影响现有项目的初始化行为
- 新增功能可能增加代码复杂性和维护成本
- 需要确保新功能与现有架构的兼容性
