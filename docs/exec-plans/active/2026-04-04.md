# 2026-04-04 Harness-Commander 初始实现计划

## Goal

- 基于现有产品规格与治理文档，完成 Harness-Commander 的首轮技术落地规划。
- 建立可扩展的 Python CLI 架构，使 `harness <command>` 成为统一入口，并为后续七类能力提供稳定分层基础。
- 优先完成可验证的最小骨架：命令入口、规划能力、计划校验能力、统一结果模型、基础文档解析与证据留痕能力。
- `./test-project` 是专门用于验证 `init -p` 的测试目录。
## Context

- 已读取 `ARCHITECTURE.md`、`docs/product-specs/harness-commander.md`、`docs/PLANS.md`、`docs/PRODUCT_SENSE.md`、`docs/QUALITY_SCORE.md`、`docs/RELIABILITY.md`、`docs/SECURITY.md`、`docs/design-docs/core-beliefs.md`。
- 当前仓库尚无 Python 源码与项目配置，适合先搭建清晰分层，再按命令能力逐步实现。
- 现有治理文档大多仍是约束模板，首轮实现需要优先把“可机器判断的规则”沉淀为代码中的校验点与统一错误输出。

## Business Logic

- Harness-Commander 的第一轮实现以统一命令协议为中心，确保不同开发者工具接入时不复制规则逻辑。
- 首轮优先覆盖 `init`、`propose-plan`、`plan-check`、`collect-evidence`，让规划、校验和证据形成最小治理闭环。
- 命令实现必须以仓库根目录为基准，并优先引用 `ARCHITECTURE.md`、`docs/PLANS.md`、`docs/RELIABILITY.md`、`docs/SECURITY.md` 与产品规格文档。
- `harness init` 必须支持 `-p` 参数，用于指定目标项目根目录；未传入时默认以当前目录执行。
- `harness init` 的行为不是覆盖初始化，而是“补齐缺失”：目录不存在则创建目录，文件不存在则创建文件，文件已存在则跳过并记录结果。
- `harness init` 的补齐目标必须覆盖根目录治理文件、`docs/` 下的设计文档、执行计划、自动生成区、业务规格与 references 小抄。
- `harness init` 对目标目录的创建范围必须严格受白名单约束，不允许额外创建 `src/`、`tests/`、`.venv/`、`venv/`、`dist/`、`build/`、`docs/generated/evidence/` 或任何未列入补齐清单的路径。
## Scope

- CLI 统一入口与命令注册机制。
- 文档感知层：读取仓库规范文件、计划文件、参考资料。
- 结果输出层：人类可读摘要与 `--json` 结构化输出。
- 首批命令：`init`、`propose-plan`、`plan-check`、`collect-evidence` 的最小可用版本。
- 测试与验证骨架：Pytest、基础单测、命令级集成验证。
- `init -p` 针对目标目录的补齐逻辑、目录创建逻辑与已存在文件跳过逻辑。
- `init -p` 针对目标目录的白名单校验与过量创建阻断逻辑。

## Steps

1. 建立 Python 项目基础设施
   - 创建 `pyproject.toml`，固定 Python 版本范围、依赖与测试配置。
   - 约定 `src/harness_commander/` 为源码根目录，`tests/` 为测试目录。
   - 约定 `./test-project` 为 `init -p` 命令测试目录；未传入 `-p` 时默认使用当前目录。
   - 选择标准库优先方案，避免首轮引入不必要依赖。

2. 设计系统分层与核心对象
   - `presentation.cli`：参数解析、命令分发、退出码控制。
   - `application.commands`：每个命令的用例编排与输入输出 DTO。
   - `domain`：命令结果模型、规则校验结果、计划语义模型、错误分类。
   - `infrastructure.filesystem`：文件读写、路径解析、目录扫描、证据落盘。
   - `infrastructure.docs`：核心文档加载、计划模板渲染、规则依赖检查。

3. 定义统一命令协议
   - 所有命令返回统一结果对象，至少包含 `command`、`status`、`summary`、`artifacts`、`warnings`、`errors`。
   - 所有写操作命令支持 `--dry-run`。
   - 所有命令支持文本摘要输出；核心命令支持 `--json`。

4. 实现首批最小能力
   - `init`：支持 `-p` 参数；补齐缺失目录与模板，目录不存在则创建，文件不存在则创建，已有文件则跳过且不覆盖。
   - `init`：补齐清单仅允许覆盖 `AGENTS.md`、`ARCHITECTURE.md`、`docs/DESIGN.md`、`docs/FRONTEND.md`、`docs/PLANS.md`、`docs/PRODUCT_SENSE.md`、`docs/QUALITY_SCORE.md`、`docs/RELIABILITY.md`、`docs/SECURITY.md`、`docs/design-docs/index.md`、`docs/design-docs/core-beliefs.md`、`docs/exec-plans/active/`、`docs/exec-plans/completed/`、`docs/exec-plans/tech-debt-tracker.md`、`docs/generated/db-schema.md`、`docs/product-specs/index.md`、`docs/product-specs/new-user-onboarding.md`、`docs/references/design-system-reference-llms.txt`、`docs/references/nixpacks-llms.txt`、`docs/references/uv-llms.txt`。
   - `init`：禁止在目标目录创建 `src/`、`tests/`、`.venv/`、`venv/`、`dist/`、`build/`、`docs/generated/evidence/` 或任何未进入白名单的目录与文件。
   - `propose-plan`：将输入需求转成 `docs/exec-plans/active/` 计划文件，包含目标、范围、验收、异常处理。
   - `plan-check`：校验计划是否引用 `ARCHITECTURE.md`、`docs/PLANS.md` 与相关规范，且具备验收和异常处理。
   - `collect-evidence`：记录命令、退出码、摘要、关键日志与时间范围。

5. 建立可靠性与安全基线
   - 错误分类为用户输入错误、环境错误、规则缺失错误、执行失败错误。
   - 日志禁止输出敏感信息，保留路径、参数摘要、失败原因与上下文。
   - 对外部命令与文件系统失败提供有限重试或明确失败摘要。

6. 编写测试与验证闭环
   - 为结果模型、计划校验逻辑、目录初始化逻辑编写 Pytest 单测。
   - 为 CLI 入口编写命令级测试，覆盖成功、失败、`--json`、`--dry-run`。
   - 运行测试与静态检查命令，确保基础骨架稳定。

## Acceptance Criteria

- AC 1: CLI 统一入口可调用 `init`、`propose-plan`、`plan-check`、`collect-evidence` 四个命令。
- AC 2: 所有命令支持统一结果结构输出，并提供 `--json` 机器可解析结果。
- AC 3: `init` 必须支持 `-p <path>` 指定目标目录；未传入时默认在当前目录执行。
- AC 4: 所有写入型命令支持 `--dry-run`，不会静默覆盖已有文件。
- AC 5: `init` 执行后，目录不存在则创建目录，文件不存在则创建文件，已有文件保持不变，只补齐缺失项。
- AC 6: `init -p ./test-project` 执行后，目标目录中必须补齐治理根文件、`docs/design-docs/`、`docs/exec-plans/`、`docs/generated/`、`docs/product-specs/`、`docs/references/` 等约定结构。
- AC 7: `plan-check` 能够识别缺少治理文档引用、缺少验收标准和缺少异常处理的计划文件。
- AC 8: `init -p ./test-project` 执行后，目标目录中不得出现 `src/`、`tests/`、`.venv/`、`venv/`、`dist/`、`build/`、`docs/generated/evidence/` 或任何未列入补齐白名单的目录与文件。

## Exception Handling

- 当核心治理文档缺失时，直接失败并输出缺失文件路径。
- 当计划文件不存在或结构不完整时，`plan-check` 返回失败结果并列出问题项。
- 当 `init -p` 指定的目标目录不存在时，先创建目录树再补齐模板文件。
- 当 `init` 发现目标文件已存在时，跳过已有文件并在结果中列出跳过项。
- 当 `init` 识别到待创建目录或文件超出补齐白名单时，必须直接失败，并输出超出项清单。
- 当证据文件或计划文件发生路径冲突时，优先生成新文件名或停止覆盖。
- 当测试或命令执行失败时，`collect-evidence` 仍保留失败证据，不伪造成功状态。
