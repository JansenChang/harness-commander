# RELIABILITY

## 这个文件是做什么的

用于规定系统在错误处理、日志记录、重试、超时和降级方面的硬性要求。

## 适合写什么

- 接口异常处理规范
- 重试策略和超时策略
- 日志字段要求和禁止项
- 告警、监控、降级和兜底方案

## 推荐用法

- AI 在写接口、任务调度、集成第三方时先参考这里
- 防止出现只实现 happy path、不处理失败路径的代码
- 把稳定性要求前置到开发阶段，而不是线上出问题后补救

## 可靠性目标

- 默认目标不是“尽量不报错”，而是“出错时结果稳定、信息充分、可追踪、可恢复”。
- 对 Harness-Commander 来说，可靠性首先体现为：
  - 错误结果合同稳定
  - 文件产物行为稳定
  - fallback 语义稳定
  - AI / 宿主模型参与时不篡改真实执行事实

## 硬性要求

### 1. 错误处理

- 所有命令必须把预期错误收敛为结构化结果，不允许直接把底层异常暴露为唯一用户信息。
- 预期错误必须具备：
  - 稳定 `code`
  - 面向用户的 `message`
  - 可定位的 `location`
  - 必要的 `detail`
- 不允许把权限错误、目录解析失败、配置缺失、输入文件缺失这类可预期问题都打成 `unexpected_error`。

### 2. 失败语义

- 文本输出、JSON 输出、落盘产物必须反映同一份失败事实。
- `warning` 与 `failure` 的语义必须清晰：
  - `warning`：流程可继续，但结果不完整或被阻断了后续动作
  - `failure`：当前命令未完成最小承诺
- fallback 不能伪装成成功：
  - 发生 fallback 时必须在结果中明确标记来源、原因和影响范围
  - 宿主模型失败后回退到 heuristic，必须留下 fallback 事实

### 3. 文件与命令稳定性

- 写入型命令必须支持 `--dry-run`，且 dry-run 的结果合同必须与真实写入路径一致。
- 任何真实写入失败，都必须返回稳定的失败原因，而不是只依赖堆栈。
- 遇到目标文件冲突时，必须要么安全避让，要么明确失败；不允许静默覆盖。

### 4. 日志与可观测性

- 日志必须帮助定位问题，不允许只输出“失败了”。
- 日志中至少应包含命令名、目标路径、关键模式、provider 或主要上下文。
- 禁止把敏感信息、长原文、密钥或不必要的用户输入直接写入日志。
- 对宿主模型调用失败，应记录 provider、阶段、失败类型，不记录敏感内容全文。

### 5. 重试 / 超时 / 降级

- 只有幂等且可恢复的动作才允许重试。
- 对宿主模型或外部 CLI 调用必须设置超时。
- 发生超时、结构化解析失败、返回空结果时，应优先降级到本地规则路径，而不是把错误结果冒充成功。
- 降级必须可解释：结果里要能看出是否降级、从哪里降级、为什么降级。

## AI / Codex 失误留痕规则

- 是的，AI / Codex 的已确认错误必须记录，但不应只写在本文件里。
- 本文件定义的是规则；具体事件要记录到：
  - `docs/exec-plans/tech-debt-tracker.md`
  - 或对应完成归档文档 / 事故复盘文档
- 每次已确认的 AI / Codex 失误，至少记录四项：
  - 错误现象
  - 根因
  - 修复方式
  - 防复发措施（测试 / 新规则 / 文档更新）
- 如果某类错误重复出现两次及以上，应把它上升为长期规则，回写到本文件或 `docs/QUALITY_SCORE.md`。

## 命令级可靠性清单

- `install-provider`
  - 目录解析失败、权限失败、CLI 未探测到、wrapper 缺失都必须返回稳定结果
- `run-agents`
  - verify 缺失 / 非 PASS / 计划校验失败必须阻断 PR summary
- `distill`
  - 宿主模型失败时必须保留 fallback 事实，不能静默改写为 heuristic 成功
- `check`
  - 阻断项、提醒项、未量化规则必须明确分层，不能混成单一“失败”

## AI 自检清单

- 我的错误路径是否有稳定错误码，而不是落成泛化异常？
- fallback 是否被清楚记录，而不是掩盖？
- dry-run 与真实写入的合同是否一致？
- 如果这次是 AI / Codex 误修，我是否把事件记到 `tech-debt-tracker` 或归档文档？
- 另一个 agent 只看结果对象，能否判断发生了什么、为什么、影响到哪里？
